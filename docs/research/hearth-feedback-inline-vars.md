# Hearth Feedback: Inline Decode Loop with ValDefs.createVar

## Problem Statement

Kindlings' jsoniter decoder generates code that boxes all primitive fields through `Array[Any]`:

```scala
// Current generated code (simplified)
val arr = new Array[Any](3)
dispatch(readKeyAsString(), arr, reader)  // arr(0) = reader.readString(null)  ← OK, String is reference
while (reader.isNextToken(',')) {
  dispatch(readKeyAsString(), arr, reader) // arr(1) = reader.readInt()         ← BOXES Int→Integer
}
new SimpleCC(
  arr(0).asInstanceOf[String],
  arr(1).asInstanceOf[Int],     // ← UNBOXES Integer→Int
  arr(2).asInstanceOf[Boolean]  // ← UNBOXES Boolean→boolean
)
```

The original jsoniter-scala generates unboxed typed locals:

```scala
var _name: String = null
var _age: Int = 0           // no boxing
var _active: Boolean = false // no boxing
// ... inline while loop ...
new SimpleCC(_name, _age, _active)
```

The gap: kindlings runs at 17M ops/s, original at 35M ops/s. The boxing accounts for most of the difference.

## What We Need from Hearth

We need `ValDefs.createVar` to work in a pattern where:

1. Multiple typed vars are created (one per case class field)
2. An additional var is created for the JSON key (`var _key: String`)
3. A `while` loop is generated inline with all vars in scope
4. A dispatch chain (if-else over field names) references the key getter and field setters
5. The constructor references all field getters
6. All of this is inside a single `ValDefs.use` scope

## Expected Generated Code

For `case class SimpleCC(name: String, age: Int, active: Boolean)`:

```scala
{
  var _name: String = null
  var _age: Int = 0
  var _active: Boolean = false
  var _key: String = null
  val _reader: JsonReader = <reader>
  if (!_reader.isNextToken('{')) _reader.decodeError("expected '{'")
  if (!_reader.isNextToken('}')) {
    _reader.rollbackToken()
    _key = _reader.readKeyAsString()
    if (_key == "name") _name = decode_field_String(_reader, config)
    else if (_key == "age") _age = decode_field_Int(_reader, config)
    else if (_key == "active") _active = decode_field_Boolean(_reader, config)
    else _reader.skip()
    while (_reader.isNextToken(',')) {
      _key = _reader.readKeyAsString()
      if (_key == "name") _name = decode_field_String(_reader, config)
      else if (_key == "age") _age = decode_field_Int(_reader, config)
      else if (_key == "active") _active = decode_field_Boolean(_reader, config)
      else _reader.skip()
    }
    if (!_reader.isCurrentToken('}')) _reader.objectEndOrCommaError()
  }
  new SimpleCC(_name, _age, _active)
}
```

Key requirements:
- `_age: Int` and `_active: Boolean` are unboxed (no `Integer`, no `java.lang.Boolean`)
- The dispatch chain references `_key` by name through `Expr.splice(keyGetter)`
- The setters (`_age = decode_field_Int(...)`) are generated by `setter(decodeExpr)`
- The `while` loop is inline in the same block as the var declarations
- No closures, no `Function0`/`Function2` allocations, no `Array[Any]`

## Macro-Level Code Pattern

```scala
// 1. Create field vars
val fieldVarDefs: List[ValDefs[(String, Expr_??, Expr[JsonReader] => Expr[Unit])]] =
  fieldDataList.map { case (fName, mappedName, decodeFn, param) =>
    import param.tpe.Underlying as Field
    ValDefs.createVar[Field](deriveZeroValue[Field], s"_$fName").map { case (getter, setter) =>
      val setFromReader: Expr[JsonReader] => Expr[Unit] = { readerExpr =>
        setter(Expr.quote(Expr.splice(decodeFn).apply(Expr.splice(readerExpr))))
      }
      (fName, getter.as_??, setFromReader)
    }
  }

// 2. Compose all field vars + key var
val allFieldVars = fieldVarDefs.foldLeft(ValDefsTraverse.pure(List.empty)) { ... }
val keyVar = ValDefs.createVar[String](Expr.quote(null: String), "_key")
val combined = allFieldVars.map2(keyVar) { case (fields, (keyGet, keySet)) =>
  (fields, keyGet, keySet)
}

// 3. Generate inline code in use scope
combined.use { case (fieldInfos, keyGetter, keySetter) =>
  val readerExpr = dctx.reader
  
  // Build dispatch chain referencing keyGetter and field setters
  val dispatch: Expr[Unit] = fieldInfos.foldRight(
    Expr.quote { Expr.splice(readerExpr).skip() }: Expr[Unit]
  ) { case ((_, mappedName, _, setFromReader), elseExpr) =>
    Expr.quote {
      if (Expr.splice(keyGetter) == Expr.splice(Expr(mappedName)))
        Expr.splice(setFromReader(readerExpr))
      else Expr.splice(elseExpr)
    }
  }
  
  // Build constructor from getters
  val fieldMap = fieldInfos.map(f => (f._1, f._2)).toMap
  val constructExpr = caseClass.primaryConstructor(fieldMap).getOrElse(...)
  
  // Generate inline loop
  Expr.quote {
    if (!Expr.splice(readerExpr).isNextToken('{')) Expr.splice(readerExpr).decodeError("expected '{'")
    if (!Expr.splice(readerExpr).isNextToken('}')) {
      Expr.splice(readerExpr).rollbackToken()
      Expr.splice(keySetter(Expr.quote(Expr.splice(readerExpr).readKeyAsString())))
      Expr.splice(dispatch)
      while (Expr.splice(readerExpr).isNextToken(',')) {
        Expr.splice(keySetter(Expr.quote(Expr.splice(readerExpr).readKeyAsString())))
        Expr.splice(dispatch)
      }
      if (!Expr.splice(readerExpr).isCurrentToken('}')) Expr.splice(readerExpr).objectEndOrCommaError()
    }
    Expr.splice(constructExpr)
  }
}
```

## What Failed Previously

We tried an intermediate approach using a runtime helper:

```scala
def readObjectDirect[A](in: JsonReader, construct: => A)(dispatch: (String, JsonReader) => Unit): A
```

This correctly used typed vars from `ValDefs.createVar`, but:
- `construct: => A` compiles to `Function0[A]` — closure allocation per decode
- `dispatch: (String, JsonReader) => Unit` — lambda allocation capturing all var references
- These closure allocations are MORE expensive than the `Array[Any]` boxing they replace
- Result: 8.6M ops/s (worse than 17.4M with `Array[Any]`)

The lesson: typed vars only help when the entire decode loop is **inline** — no runtime helper, no closures.

## How to Reproduce in Hearth (Test Case)

A minimal hearth test that validates the pattern:

```scala
// In hearth test suite
test("ValDefs.createVar vars accessible in while loop with spliced dispatch") {
  val result: Expr[Int] = {
    val intVar = ValDefs.createVar[Int](Expr(0), "_count")
    val strVar = ValDefs.createVar[String](Expr.quote(null: String), "_key")
    
    intVar.map2(strVar) { case ((countGet, countSet), (keyGet, keySet)) =>
      (countGet, countSet, keyGet, keySet)
    }.use { case (countGet, countSet, keyGet, keySet) =>
      val dispatch: Expr[Unit] = Expr.quote {
        if (Expr.splice(keyGet) == "increment")
          Expr.splice(countSet(Expr.quote(Expr.splice(countGet) + 1)))
        else ()
      }
      
      Expr.quote {
        val items = List("increment", "skip", "increment", "increment")
        val iter = items.iterator
        while (iter.hasNext) {
          Expr.splice(keySet(Expr.quote(iter.next())))
          Expr.splice(dispatch)
        }
        Expr.splice(countGet) // should be 3
      }
    }
  }
  assertEquals(eval(result), 3)
}
```

This test verifies:
1. Multiple `createVar` declarations compose via `map2`
2. Var getters/setters work inside `while` loops
3. Spliced dispatch chains reference vars from the enclosing `use` scope
4. The generated code uses unboxed `Int` (no `Integer` boxing)

Additionally, verify with bytecode inspection that the generated code contains:
- `iload`/`istore` for the Int var (not `invokevirtual Integer.intValue` / `invokestatic Integer.valueOf`)
- No `Function0`, `Function1`, `Function2` allocations in the loop body

## Guard Against Accidental Fix

The test case should specifically verify that:

1. **The var is truly unboxed**: Add an assertion that checks the bytecode doesn't contain boxing instructions for the specific codec method. A test that only checks correctness (round-trip encode/decode) would pass even with boxing.

2. **The pattern works for mixed types**: Test with a case class that has BOTH primitive fields (Int, Boolean, Double) AND reference fields (String, Option, List). The primitives must be unboxed while references remain as-is.

3. **The pattern works with nested types**: Test with `Person` which has `List[Address]` — the Address decoder is a cached def, not a built-in type. The var for `addresses` should be `var _addresses: List[Address] = null` (reference type, no boxing concern), while `var _age: Int = 0` remains unboxed.

4. **NOT fixed by semiEval alone**: If someone adds a semiEval optimization that happens to improve the numbers, but doesn't eliminate boxing, the bytecode check will catch it. The fix must eliminate the `Array[Any]` pattern entirely, not just reduce the number of config checks.
